# 基于知识图谱的职位推荐系统 — 项目说明文档

## 1. 选题背景与意义

### 1.1 问题背景

随着人工智能行业的快速发展，AI 相关岗位的招聘需求日益增长。求职者在面对前程无忧（51job）、Boss 直聘、拉勾网等平台上海量分散的招聘信息时，往往面临以下痛点：

- **信息过载**：一个岗位关键词下可能有数百条结果，人工筛选效率低下
- **维度单一**：传统关键词搜索无法表达"我会 Python+PyTorch，想去上海，月薪 15k 以上"这类复合需求
- **关联缺失**：求职者难以发现技能相近但名称不同的岗位，也难以横向比较同城、同行业岗位

### 1.2 解决思路

本项目引入**知识图谱**（Knowledge Graph）技术，将职位、公司、技能、行业等实体及其关系组织为一张"知识网"，从而：

- 通过图关系实现**多维度复合检索**（技能 + 城市 + 行业 + 薪资）
- 基于用户画像在图谱中做**路径遍历与加权评分**，实现智能推荐
- 通过图可视化帮助求职者**直观理解**岗位与技能/行业/公司的关联

### 1.3 项目目标

构建一套完整的知识图谱职位推荐系统，覆盖从数据采集到前端展示的全链路，具体目标包括：

- 爬取并清洗 51job 上 8 大城市的 AI 相关岗位数据
- 构建包含 Job、Company、City、Skill、Benefit 等节点的知识图谱
- 提供基于用户画像的智能推荐接口
- 提供相似岗位推荐接口
- 交付可交互的前端页面，包含推荐结果和图谱可视化

## 4. 各任务详细说明

### 任务 1：数据采集

#### 负责小组：数据采集与预处理

#### 技术方案

- **爬虫框架**：Selenium + Chrome（headless=False 可视化模式）
- **目标站点**：前程无忧 51job（https://we.51job.com/pc/search）
- **数据提取方式**：通过 Chrome DevTools Protocol 执行 JavaScript，直接读取 DOM 中的 `sensorsdata` JSON 属性，跳过页面渲染等待
- **搜索策略**：覆盖北京、上海、深圳、广州、杭州、成都、南京、武汉 8 大城市，关键词包括"人工智能""机器学习""深度学习""算法工程师""AI 工程师"
- **分页控制**：每页 50 条，最多爬取 10 页/城市/关键词

#### 主要难点与解决方案

**难点一：51job 反爬机制**

51job 通过以下方式检测自动化爬虫：

- `navigator.webdriver` 标识（Selenium 默认暴露为 `true`）
- `navigator.plugins` 和 `navigator.languages` 指纹
- 滑动验证码拦截

**解决方案**：

```python
# 隐藏 webdriver 标识
chrome_options.add_argument("--disable-blink-features=AutomationControlled")

# JS 注入覆盖指纹
driver.execute_cdp_cmd("Page.addScriptToEvaluateOnNewDocument", {
    "source": """
        Object.defineProperty(navigator, 'webdriver', {get: () => undefined});
        navigator.plugins = [1, 2, 3];
        Object.defineProperty(navigator, 'languages', {get: () => ['zh-CN', 'zh']});
    """
})
```

**难点二：滑动验证码处理**

51job 的滑动验证码无法通过程序自动破解。

**解决方案**：爬虫以**可见模式**运行，检测到验证码时弹出浏览器窗口，提示用户在界面上手动滑动，完成后爬虫自动继续抓取。

**难点三：薪资格式多样**

原始数据中薪资格式混乱，例如：

- `15-25k`（常见）
- `1.5-2.5万/月`（需转换）
- `8000-10000元/月`（需转换）

**解决方案**：使用正则表达式统一解析后转换为"千/月"为单位存储。

### 任务 2：数据清洗与去重

#### 负责小组：数据采集与预处理

#### 技术方案

- **字段标准化**：将原始字段（`jobName`、`companyName`、`jobAreaString` 等）统一映射为规范字段
- **去重策略**：以 `MD5(jobName|companyName)` 作为唯一键，去除完全重复的记录
- **三层数据文件**：
  - `datasets/raw/jobs_raw.json` — 原始数据，保留 sensorsdata 和详情链接
  - `datasets/interim/jobs_cleaned.json` — 清洗后数据
  - `datasets/processed/jobs.json` — 去重后最终数据，供图谱使用

#### 主要难点与解决方案

**难点一：城市字段含区级信息**

原始 `jobAreaString` 如"上海·浦东新区"，需截取到城市级别。

**解决方案**：按 `·` 或 `|` 分割字符串，取第一段作为城市名。

**难点二：经验/学历字段不规范**

原始数据中经验字段如"3-4年经验"，需归一化为"3-4年"。

**解决方案**：建立归一化映射表，处理"无要求""在校/应届"等边界情况。

**难点三：字段缺失**

部分记录缺少岗位名称或公司名称，直接过滤。

### 任务 3：实体抽取

#### 负责小组：知识图谱构建

#### 技术方案

在 `job_kg/taxonomy.py` 中实现基于规则的实体抽取引擎，从以下三个来源提取实体：

1. **jobTags**：显式标签，逗号/顿号/斜杠分隔
2. **jobName**：岗位名称中的技能关键词
3. **jobDescribe**：岗位描述文本，补充召回

抽取三类实体：

| 实体类型    | 来源         | 示例              |
| ----------- | ------------ | ----------------- |
| **Skill**   | 技能别名映射 | pytorch → PyTorch |
| **Benefit** | 福利关键词库 | 五险一金、年终奖  |
| **Keyword** | 剩余有效标签 | 计算机视觉、NLP   |

#### 主要难点与解决方案

**难点一：技能别名归一化**

同一技能有多种写法，如 `pytorch`/`PyTorch`/`pytorch框架`、`Python`/`python`/`py`。

**解决方案**：建立技能别名映射表，包含 25 类规范技能名称和 100+ 变体，全部映射到规范写法（如 PyTorch）。

**难点二：福利与关键词区分**

"计算机"一词可能是技能（计算机视觉），也可能是停用词（计算机相关专业）。

**解决方案**：建立 27 项停用词库（计算机、数学、英语、技术交流等），优先匹配技能别名，不在别名中且不在停用词中的剩余标签归为 Keyword。

**难点三：描述文本中的隐式技能**

岗位描述中可能出现"熟悉 TensorFlow/PyTorch"等技能提及。

**解决方案**：对描述文本做正则匹配，提取 TensorFlow、PyTorch、机器学习等高频关键词作为补充。

### 任务 4：Neo4j 图谱建模与同步

#### 负责小组：知识图谱构建

#### 技术方案

**节点建模**（9 类节点）：

| 节点       | 说明     | 示例            |
| ---------- | -------- | --------------- |
| Job        | 职位     | "AI 算法工程师" |
| Company    | 公司     | "字节跳动"      |
| City       | 城市     | "上海"          |
| Industry   | 行业     | "互联网"        |
| Degree     | 学历要求 | "本科"          |
| Experience | 经验要求 | "3-5年"         |
| Skill      | 技能     | "Python"        |
| Benefit    | 福利     | "五险一金"      |
| Keyword    | 关键词   | "机器学习"      |

**关系建模**（9 类关系）：

```
(:Job)-[:POSTED_BY]->(:Company)      # 职位由公司发布
(:Job)-[:LOCATED_IN]->(:City)        # 职位位于城市
(:Job)-[:IN_INDUSTRY]->(:Industry)   # 职位属于行业
(:Job)-[:REQUIRES_DEGREE]->(:Degree) # 学历要求
(:Job)-[:REQUIRES_EXPERIENCE]->(:Exp)# 经验要求
(:Job)-[:REQUIRES_SKILL]->(:Skill)   # 技能要求
(:Job)-[:HAS_BENEFIT]->(:Benefit)    # 福利待遇
(:Job)-[:HAS_KEYWORD]->(:Keyword)    # 关键词标签
(:Company)-[:BELONGS_TO]->(:Industry)# 公司属于行业
```

**同步策略**：使用 `MERGE` 原子操作，先创建唯一性约束，再清空旧图，执行批量 UNWIND 导入。

#### 主要难点与解决方案

**难点一：Neo4j 服务部署**

本地开发环境中 Docker Desktop 网络路径问题导致 Neo4j 容器端口映射失效。

**解决方案**：切换使用 **Neo4j Aura** 云数据库，通过 `neo4j+s://` 协议连接，避免本地容器运维负担。

**难点二：数据库认证**

Neo4j Aura 首次连接需要等待实例初始化，且认证凭证格式与本地不同。

**解决方案**：配置 `NEO4J_URI=neo4j+s://xxx.databases.neo4j.io`、`NEO4J_AUTH=user/password`，等待 60 秒后再发起连接。

### 任务 5：FastAPI 接口开发

#### 负责小组：系统构建

#### 技术方案

在 `job_kg/api.py` 中实现 9 个 REST 端点：

| 方法 | 路径                     | 功能                               |
| ---- | ------------------------ | ---------------------------------- |
| GET  | `/`                      | 返回前端 HTML 页面                 |
| GET  | `/health`                | 健康检查                           |
| GET  | `/stats`                 | 图谱统计（岗位数/公司数/技能数等） |
| GET  | `/jobs`                  | 条件查询岗位列表                   |
| GET  | `/jobs/{job_id}`         | 获取单个岗位详情                   |
| GET  | `/jobs/{job_id}/graph`   | 获取岗位一跳子图                   |
| GET  | `/jobs/{job_id}/similar` | 相似岗位推荐                       |
| POST | `/recommend/profile`     | 基于用户画像推荐岗位               |
| GET  | `/skills/top`            | 高频技能排行                       |

所有接口使用 Pydantic 模型定义请求/响应 schema，由 FastAPI 自动生成 OpenAPI 文档。

#### 主要难点与解决方案

**难点：Neo4j 与 Python 对象的映射**

Neo4j Cypher 查询返回的是原生字典，需手工映射为 Pydantic 模型。

**解决方案**：在 `repository.py` 中封装 `_record_to_summary()` 等私有方法，统一将 Cypher 结果转换为类型安全的 Pydantic 模型。

### 任务 6：前端页面开发

#### 负责小组：系统构建

#### 技术方案

单页 HTML 应用（`frontend/index.html`），无框架依赖：

- **左侧边栏**（360px）：用户画像输入表单 + 岗位 ID 图谱查看器
- **主内容区**：推荐结果卡片列表
- **底部区域**：ECharts 力导向图（force-directed graph）

核心交互：

1. 用户输入技能、城市、薪资 → `POST /recommend/profile`
2. 展示推荐结果卡片，含匹配得分和推荐理由
3. 点击"查看图谱" → `GET /jobs/{id}/graph` → 渲染 ECharts 子图

#### 主要难点与解决方案

**难点：图谱节点布局混乱**

初始布局时所有节点堆叠，无法直观看出岗位与周边实体的关系。

**解决方案**：ECharts `force` 布局配置 `repulsion: 180`（节点斥力）和 `edgeLength: 110`（边长），Job 节点设置 `symbolSize: 58`（较大），其他节点 `symbolSize: 42`（较小），自动形成以岗位为中心的辐射布局。

### 任务 7：推荐算法实现

#### 负责小组：系统构建

#### 技术方案

##### 画像推荐

根据用户画像（技能、城市、行业、薪资）计算岗位匹配得分，在 Neo4j 中通过 Cypher 实现：

```
score = 匹配技能数 × 4.0
      + 匹配关键词数 × 1.5
      + 城市命中 × 3.0
      + 行业命中 × 2.5
      + 薪资达标 × 1.0
```

过滤条件：总分 > 0，且满足（无技能要求 OR 有技能匹配 OR 分数 ≥ 4.0）

排序：`ORDER BY score DESC, salary_mid DESC, skills_count DESC`

##### 相似岗位推荐

给定一个岗位 ID，基于图关系寻找相似职位：

```
score = 共享技能数 × 3.0
      + 同城市 × 2.0
      + 同行业 × 2.0
      + 同公司 × 1.0
```

##### 推荐理由生成

每条推荐结果附带人类可读的推荐理由：

| 匹配条件 | 推荐理由                   |
| -------- | -------------------------- |
| 技能匹配 | "匹配技能 Python, PyTorch" |
| 城市命中 | "城市匹配 上海"            |
| 行业命中 | "行业匹配 互联网"          |
| 薪资达标 | "薪资满足 15-25k"          |
| 学历兼容 | "学历匹配 本科"            |

#### 主要难点与解决方案

**难点：权重参数选择**

技能、城市、行业等维度的权重比例直接影响推荐效果，难以通过离线评测确定。

**解决方案**：参考业界推荐系统经验（技能匹配最重要），设定初值（4.0/3.0/2.5），通过返回得分明细（`score` 字段）和推荐理由（`reasons` 字段）提升可解释性，便于后续人工调优。
